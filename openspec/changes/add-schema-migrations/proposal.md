# add-schema-migrations

**Phase:** beta-release-readiness / Phase 2 (Serious gap S-1)

## Problem

The entire Postgres schema is one inline raw SQL string constant applied via
`sqlx::raw_sql` at startup. There is no version tracking, no rollback capability,
and no way to know what schema version is running in production. Any schema change
during the beta period (e.g. adding a column to support Cedar schema storage) is
unversioned and unauditable.

## Solution

Adopt sqlx versioned migrations:

1. Create `crates/flint-gate-core/migrations/` directory
2. Extract the current DDL into numbered `.sql` files:
   - `0001_initial_schema.sql` — all `CREATE TABLE IF NOT EXISTS` DDL
   - `0002_alter_api_keys.sql` — the existing `ADD COLUMN IF NOT EXISTS` statements
3. Replace `sqlx::raw_sql(SCHEMA_SQL)` with `sqlx::migrate!("migrations")` macro
4. Remove the `SCHEMA_SQL` string constant
5. Run `cargo sqlx prepare` to generate `.sqlx/` offline query cache for CI

## Files to change

- `crates/flint-gate-core/src/db/mod.rs` — replace raw SQL migration with sqlx migrate
- `crates/flint-gate-core/migrations/0001_initial_schema.sql` (new)
- `crates/flint-gate-core/migrations/0002_alter_api_keys.sql` (new)
- `Cargo.toml` — add `sqlx-cli` to dev dependencies (optional; can also use npx)
- `.sqlx/` directory (generated by `cargo sqlx prepare`)

## Notes

- Existing deployments that already have the `IF NOT EXISTS` schema will see no
  change — sqlx migrations track applied migrations in `_sqlx_migrations` table
- The first `cargo migrate` on an existing DB will mark all prior migrations as
  applied (they are idempotent)
